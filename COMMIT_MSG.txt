Fix: CORS headers now injected on every response including error handlers

Problem: After deploying, the browser reported CORS errors despite the
CORSMiddleware being configured correctly. Two issues:

1. FastAPI's built-in CORSMiddleware does NOT add CORS headers to
   responses that come from exception handlers (500, 422, etc.) —
   only normal route handlers get the headers. So when the reviews
   endpoint threw a 500 (reviews table doesn't exist yet), the
   response was missing Access-Control-Allow-Origin and the browser
   rejected it.

2. The outer ASGI wrapper was only handling OPTIONS preflight at that
   layer — non-OPTIONS requests were passed through to FastAPI
   without the CORS injection fallback.

Fix:
- The ASGI wrapper now also injects CORS headers on non-OPTIONS
  responses, but only when the CORSMiddleware hasn't already added
  them (deduplication check via has_cors)
- This ensures that error responses (5xx, 4xx) and successful
  responses all have consistent CORS headers

Verified locally with httpx:
- All response types (200, 204, 404) now have ACAO headers
- OPTIONS preflight handled at ASGI layer with proper headers
- No header duplication with CORSMiddleware
